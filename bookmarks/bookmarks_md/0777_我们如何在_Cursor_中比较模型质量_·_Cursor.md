# 我们如何在 Cursor 中比较模型质量

Cursor Blog · 研究

开发者正让编程智能体承担更长、更复杂的任务，这些任务跨越多个文件、工具和步骤。随着这些请求的范围不断扩大，用于衡量智能体性能的评测也需要随之演进。

在 Cursor，我们采用线上与线下结合的混合评测流程，让我们对模型质量的理解始终与开发者的真实工作方式保持一致。

## CursorBench

线下部分使用 CursorBench，这是我们基于工程团队真实 Cursor 会话构建的内部评测套件。由于任务来自真实的 Cursor 用量，而不是公开代码仓库，CursorBench 不仅更能区分不同模型，也比公开基准更贴近开发者的真实结果。

CursorBench 衡量智能体性能的多个维度：解决方案正确性、代码质量、效率和交互行为。正确性得分与完成所用 token 数中位数的关系图，捕捉了影响可用性的算力/延迟权衡。右上角代表理想的智能体质量：以最低成本实现最高性能。

我们还通过在真实流量上进行的受控分析来补充 CursorBench。这些线上评测能发现线下评测套件遗漏的回归问题。

## 公开基准的局限性

一个好的基准需要能够区分那些在实际使用中表现不同的模型，同时与开发者对这些模型的真实体验相一致。公开的离线评测存在困难：

1. **一致性问题**：大多数 SWE 基准仍然聚焦于修复缺陷任务。Terminal-Bench 更强调广义的谜题式任务。这些任务与开发者要求智能体完成的编程工作并不契合。

2. **评分问题**：许多公开基准任务假设只有一小组正确解法，但大多数开发者请求存在多种有效方案。

3. **污染问题**：SWE-bench 等从公开代码库中抽取任务，这些任务最终会进入模型训练数据。OpenAI 已完全停止报告 SWE-bench Verified 的结果。

## 构建 CursorBench

我们使用 Cursor Blame 为 CursorBench 收集任务，它可以将已提交的代码追溯到生成该代码的智能体请求。这为我们提供了开发者查询与标准答案解决方案的自然配对。

CursorBench-3 任务涉及的代码行数，明显多于 SWE-bench Verified、Pro 或 Multilingual 中的任务。任务也更贴近开发者与智能体交流时那种信息不充分、且常带歧义的表达方式。

## CursorBench 能更好地区分模型

随着公开基准日益趋于饱和，CursorBench 在前沿水平上能更好地区分模型。在某些情况下，Haiku 这样的模型可以媲美甚至超过 GPT-5。CursorBench 分数与在线评估结果一致。

## 下一代评测套件

我们预计在未来一年里，绝大多数开发工作将转向由长时运行智能体来完成。我们需要找到降低评测成本的方法，解决与外部服务交互类任务的可复现性问题，并缩小离线评估与开发者体验之间的差距。
