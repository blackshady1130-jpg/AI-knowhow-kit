# 展望2026年的Agent应用层演进

Original 孔某人 | 孔某人的低维认知

## 1、Coding

Coding 能力对于 Agent 类似于使用电脑的技能对于打工人——普及是普及的，但 Coding 能力对于大部分场景来说只是手段，不是目的。

## 2、Context Window 进展缓慢导致 Agent Harness 进行适配

虽然很多模型号称有 1M token window，但考虑到成本和效果，很多都仍然只是在用 200k-300k token。成本是最主要因素。

Long Horizon Task 开始逐步变得可行，对 Context 有较高需求，但目前 Agent Harness 常用的对话压缩和 Memory 方案都太过粗糙成为瓶颈。

这轮 GenAI 应用层创业最重要的选择之一：是基于当下技术能力先做到 PMF，还是等待？最佳路径可能是持续的平衡和过程中的不断规划。

## 3、Auto Research

绝大部分 Deep Research 类产品实际应该叫 Deep Search。Auto Research 是 Long Horizon Task 中比较典型的一类应用。

Andrej Karpathy 也进行了类似的尝试并完整公开了实践方式（就像 vibe coding 这个词也是他提出的）。

在 Auto Research 范式之前，AI Coding 并不能替代算法岗的工作，只是替代了接明确需求的开发类岗位。Auto Research 类范式的逐步可行标志着算法岗的工作也可以被替代。

Karpathy 在他的 autoresearch 项目中开篇写道：曾几何时，前沿 AI 研究还是由"肉体计算机"完成的：在吃饭、睡觉、享乐之余，偶尔还要通过声波互联，在名为"组会"的仪式中进行同步。那个时代早已一去不复返。

## 4、Long Horizon Task 与 Workspace Management

持续对于一个 Workspace/Context 的持续更新和工作可以视为一类 LHT。目前 LLM 模型还不能很好地用好外部 Memory 文件系统。

OpenClaw 除了在 Memory 方案上做了复杂设计外，在 LHT 方面没有任何特殊优化。复杂 Memory 范式本身大概率有问题——好的 Memory 方案大概率不是人设计出来的，而是 auto research 做出来的。

## 结语

站在应用层使用者的视角，LLM 模型方面 Q1 只能用乏力来形容。Q2 应该有一些更大进展，更本质的进展可能得等到 Q3-Q4。
