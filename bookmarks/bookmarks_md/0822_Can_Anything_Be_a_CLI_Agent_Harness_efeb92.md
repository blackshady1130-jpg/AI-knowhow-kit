# Can Anything Be a CLI?｜当软件开始长出给 Agent 用的Harness

在今天这篇文章发出的时候，飞书CLI开源了。有人说，这代表着越来越多主流厂商拥抱把自己的业务变成Agent随时可以调用的服务；有人说，这代表了工作/效率类Agent的前沿落地，通用AI助理越来越不被过往数字世界的桎梏限制。嘉宾 Yuhao 是港大 PhD ，导师是黄超老师。他研究 GUI/CUA 方向多年，代表作有：Aria-UI，GTA1, UltraCUA，工作涉及到agent模型训练、数据、环境和测评多个方面。近期他开始专注于一个新项目：CLI anything。CLI anything 在开源社区获得越来越多赞誉（不到两周就收获超 2 万 stars）的背后，也代表着开发者，乃至 AI 从业者们对于上述强烈的 CLI 改造意愿和动力。在这个访谈中，我们聊了：现有的软件服务的“四个象限”该怎么划分：mock 软件和 CLI anything、GUI automation 和官方 API（现在官方开放CLI的热情与日俱增）。如何判断服务适不适合CLI改造。如果现有软件不开放CLI，有什么办法将其改造，这与 hack 一个软件有什么区别。这个话题又与最近 Anthropic、Kapathy重复提及的 Harness 哲学息息相关，在 LLM 即 Agent的今天，我们如何做好 Harness，应当如何思考。Yuhao 的出发点很简单：让 AI agent 替代人类在各行各业中的工作流，不管是局部替代，还是端到端全流程接管。他的观点是，如果现存软件还没有做好CLI改造的准备，起码他可以提供外部可触达的接口和更好用的自身改造工具。这个命题背后也涉及哪些数据会再次让模型受益、 自进化self-eval以及软件的未来。今天是最合适讨论CLI Anything的时间点，相信半年到一年内，就有越来越多“开源社区试图使用CLI Anything的服务” 都会直接开放自己的 CLI 端口，这是一个越来越触手可及的未来。目录：一、为什么从 GUI 转向 CLI二、四个象限：理解软件与 Agent 的交互方式三、哪些服务适合 CLI？哪些不适合？四、CLI Anything 的工程实践：7 步流程与 Harness 哲学五、REPL 与 subcommand：有状态交互的价值六、为什么 Agent 还不能驾驭复杂黑盒七、从 OSWorld 看一个有意思的对称性八、自进化与 Eval 的难题九、软件的未来一、为什么从 GUI 转向 CLI两年前，GUI agent 是一个很有吸引力的方向。理论上，只要模型能像人一样操作屏幕，就能接管几乎所有给人用的软件，不需要软件主动开放任何接口。但做了很长时间之后，Yuhao 越来越清晰地看到 GUI agent 的几个根本性瓶颈。第一是脆弱性。GUI agent 高度依赖屏幕 grounding 和 planning 能力，而这两件事在模型里是深度耦合的，很难单独提升其中一个。你很容易把模型的 grounding 能力训得更好，但一旦想让它 plan 做得更好，几乎没办法定位到底哪里出了问题——大多数时候根源是 world knowledge 不足，而这种知识又很难迁移到不同软件上。第二是稳定性差。对于那些需要高频、稳定执行的任务，GUI agent 是一个比较灾难的方案：窗口布局改了、设备换了、软件更新了，都会带来明显的适配问题。第三是整个过程更像一个黑箱。多模态 planning 和 grounding 耦合在一起，让模型训练变得难以控制。与此同时，coding 飞轮转得越来越快，模型能力越来越强，Harness 的概念也越来越火。Yuhao 的判断是：CLI 作为一种更底层、更稳定的接口形式，现在是把软件迁移给 agent 使用的合适时机。二、四个象限：理解软件与 Agent 的交互方式谈到 CLI anything 之前，有必要先把几种主流的软件交互方式放在一起梳理清楚。Yuhao 用一个四象限的框架来理解这件事。上下轴是交互层面的区分：上方是基于 GUI 的方案，下方是基于 coding 或命令行的方案。左右轴则是一个有意思的划分——是否掌握源代码。左侧（mock 软件和 CLI anything）都要求对源码有较强的掌控性，右侧（GUI automation 和官方 API）则是在不拥有源码的情况下使用软件。这四个象限分别是：1. GUI Automation：让模型直接操作真实的 GUI 界面，点击、拖曳、截图。优点是通用性极强，理论上所有给人用的软件都能操作；缺点就是上文说的那些。2. Mock 软件：制造一个仿真的假软件环境，专门用来训练 GUI agent。这条路在大厂的多模态 agent 部门比较流行，本质上是承认真实软件的 API 接口不够友好，所以用仿真环境代替。3. 官方 API：厂商主动把自己的服务暴露给外部调用，稳定、高效、原生，但依赖厂商愿意开放，且 API 文档往往是散列状的，需要额外做渐进式披露（progressive disclosure）的工程处理。现在也有不少厂商开始 CLI 改造。4. CLI anything：Yuhao 把它定位为和 API 同一侧，但更进一步——在软件已有接口不足的情况下，主动扫描软件、理解它的后端逻辑，然后帮它生成一套完备的 CLI 接口。目前社区已经有一些实践：有人把 Google Notebook LM 包装成了 CLI 格式（完全基于 Google 暴露的官方接口实现），也有人包装了 Zoom（基于 Zoom 官方的基础 API）。另外还有一条平行的路线，是基于 Web 或 Electron 应用的自动化测试技术来包装商业软件，比如把小红书、淘宝、视频网站的前端接口扫出来再封装成 CLI（OpenCLI 项目在开源社区有很好的声量和贡献）。这条路能覆盖更多商业服务，适应性强，但天然受制于厂商——如果厂商主动修改接口来屏蔽这种方式，稳定性就会受影响。CLI anything 选择的是更底层的路线，从源码或真实后端入手，转换出来的接口稳定性更高，不会随着时间推移出现大的变化。CLI 相比于普通 API 的一个关键优势是：Unix 几十年前定下的规范让 CLI 天然具备渐进式披露的结构——有 help 命令、有分层子命令、有标准化的输出格式，agent 可以非常自然地用它来探索和操作软件。三、哪些服务适合 CLI？哪些不适合？Yuhao 提出了一个四类划分，依据不是技术实现，而是服务交付的后果。第一类：私有数据驱动的服务这类服务的核心价值在于它积累了大量用户生产的私有数据，比如 Boss 直聘（工作信息）、小红书（真实用户的探店体验）、Twitter。不管是人还是 agent，想用这些服务，本质上都是在消费这里沉淀的私有数据。这类服务的 CLI 化价值在于让 agent 能更高效地检索和使用这些数据。第二类：触发物理世界交付的服务美团、Uber、顺丰这类服务，数字端的操作其实很简单，往数据库里写一条记录，广播出去，让各方接到并行动。但它的重头戏在于最后一公里——有一个物理实体真实地为你送来外卖、叫来车、送达包裹。CLI 只能完成前半段，后半段依然需要物理世界的介入。第三类：需要法人或真实主体背书的服务这类服务在数字层面上是可以完全自动化的，但现实世界的合规框架要求背后必须有一个真实存在的法人或个人作为责任主体。银行转账、投行报告、合同签署都属于此类。agent 可以完成大部分工作，但那个“最后需要人存在在那里”的护城河依然在。第四类：依赖积淀已久的专业黑盒的服务这是 Yuhao 认为最容易被忽视的一类。让 agent 处理视频，最后它还是会用 FFmpeg；让它做 3D 设计，它还是离不开 Blender 的渲染引擎；CAD 软件底层有一个几十年积累下来的几何求解器，agent 没有必要从头重写一个。这类“暗箱算力”是前几十年的软件工程师沉淀下来的护城河，agent 其实在无感地利用它们，只是大家有时候不容易发现。四、CLI Anything 的工程实践：7 步流程与 Harness 哲学Yuhao 设计了一个 7 步的 pipeline，但他强调整个框架刻意做得很 general——本质上只做两件事。第一件事：拆解软件，搭建完备的 CLI 接口让 agent 自适应地找到软件真正的 backend，理解它的底层数据组织方式。比如 LibreOffice 的底层其实是一系列 XML 文件，你只要能完整操作这些 XML，就能复现所有人类在 GUI 上能做的事。找到这个“中间形式”之后，再基于它搭建一套完备的、有状态的 CLI 接口。第二件事：做完整的端到端测试不只是单元测试，端到端测试的占比要和单元测试持平。这是为了保证交付的接口真的能满足人类使用这个软件的完整期待。在 Harness 的哲学上，Yuhao 的取向很清晰：不过度干预，不要太强的预设，让 agent 自己去 figure out 细节。他提到社区里有一些 PR 建议把超过 100KB 的文件跳过不读，或者对读文件的 token 数量加硬限制。Yuhao 把这些 PR 全部打回了，因为他认为这些是模型目前的局限，不是 Harness 本身应该去编码的偏见——换一个更强的模型，这些问题可能就不存在了。关于 Harness 的层次，Yuhao 把它分成三层：最底层是 Prompt。大家最近习惯把 prompt engineering 和 harness engineering 分开讲，但他的观察是，Claude Code 本身的系统 prompt 就有七八千字，这一层是基础，不能被忽略。他提到 X 上有一个实验：只给模型写一段很 general 的 harness 提示词，只提供 Python 和 bash 两个工具，在 SWE-bench 上也能拿到接近 90% 的分数。中间层是 工具集设计。原则是：工具数量尽量少，工具之间的 overlap 尽量小，但加起来覆盖的操作空间要尽量大。Unix 文件加字节流的这套设计，加上 web search，其实已经能构成相当完备的工具集。此外，让 agent 主动向用户提问也很重要——很多时候 instruction 本身是模糊的，agent 提问的能力能帮助把需求厘清。更高层是 工程范式，比如渐进式披露、丢掉 RAG 让 agent 自己决定怎么 search 等。这些是更 philosophy level 的东西，Claude Code 的 foundation engineer 在推动，大家跟进就好。五、REPL 与 Subcommand：有状态交互的价值CLI anything 的核心交互形式是 REPL（Read-Eval-Print Loop）。Yuhao 说他一开始设计这件事时其实不知道这个概念，只是想给任何软件都做出“Claude Code 那种感觉”的交互，后来才发现这就是 REPL。REPL 最大的价值在于“带状态”。打开一个 session 之后，所有操作默认都和这个 session 关联，不用每次都重新 specify 上下文。很多专业软件本来就是这么工作的——你开一个项目，在里面持续编辑，最后以它自己的格式存档（比如视频剪辑软件的 MLT 格式，或者 LibreOffice 的 XML 格式）。而 REPL 里的状态展示也很有价值。就像 Claude Code 会在下方列出哪些 subagent 在跑、在做什么任务，剪辑软件可以直接列出当前时间线上有哪些滤镜，让 agent 不用每次都 list 一遍再确认。相比之下，Subcommand 更适合做单点的操作——一次渲染、一次保存。如果要在里面完成所有复杂工作，需要手动 specify 每一步的状态，没有 REPL 那么直接。两者并不是替代关系，而是各有适用场景。六、为什么 Agent 还不能驾驭复杂黑盒一个自然的问题是：coding agent 已经这么强了，为什么还是很难自己操作 Blender、Autocad 这类复杂软件？Yuhao 从三个层面来解释。Pre-training 数据的占比问题。 人类发展到现在，大型专业软件的开源代码本来就远少于普通代码。游戏引擎、CAD 软件——能叫得出名字的开源项目就那几个，agent 学习的来源极为有限。软件本身的 agent-friendliness 问题。 大多数专业软件在开放程度上是有限的，它们的 API 是以原始代码实现的形式开放给模型学习，而不是以“怎么操作它”的轨迹形式。这两件事之间有明显的差距。训练目标的问题。 目前 agent 的训练目标并不是“解构和重建这些黑盒”，从黑盒中拿到 reward 的路径更长、更不直观。这造成了当前的 bottleneck。在 review 了很多软件后端之后，Yuhao 发现一个规律：CLI anything 能成功落地的 case，往往是后端本身设计得足够优秀的软件。LibreOffice 用 XML 作为中间格式，GIMP（开源版 Photoshop）和 Blender 都以 Python 作为脚本层连接底层的 C 编译产物——这些都是清晰的、可以被 agent 接管的接口。Blender 虽然是开源软件，但在工业设计领域举足轻重，它对 Python 特性的支持相当完备，不仅可以直接把 Python 接口包装成 CLI，甚至可以用 Python 专门为 Blender 造训练数据，直接训出一个能操作它的专用 agent。反面例子是那种把状态直接存在前端 UI 组件里的耦合架构——比如提交一个表单的状态完全依赖于某个 Qt 组件，agent 要处理这种情况就需要先把前后端解耦，工作量大很多。这种问题在一些商业软件里也存在，处理方式是让 agent 重新 hack 一套新的 CLI 接口，同时把状态从 Qt 组件里解绑出来，再同时绑定到两个前端上去。七、从 OSWorld 看一个有意思的对称性在 GUI agent 社区工作多年后，Yuhao 注意到一个有点别扭的地方。OSWorld 是我的fellow Tianbao 等人做的 GUI agent 评测框架。它的检查器，也就是评判 agent 有没有完成任务的逻辑，本身是用 coding 来实现的——agent 有没有生成对的 Word 文档、对的 PPT、对的视频，都是通过把这些产物转换成代码来判断的。这让 Yuhao 想到 Jason Wei（OpenAI）和姚顺雨一直在讨论的“验证不对称性”概念：有些任务验证起来容易，但生成起来难。Jason Wei 在 BrowseComp 这个 benchmark 里用到了这个思路——从难以找到但容易验证的问题入手，来衡量 agent 的 browsing 能力。OSWorld 里其实也有类似的结构，只是方向反了过来。既然 agent 能从 coding 来验证这个产物，那从另一侧看，它其实也可以直接从 coding 来创建这个产物——这正是 CLI anything 在做的事。这引出了一个更深的问题：对于 GUI agent 的训练，这种“人为制造不对称性”是不是必要的？Yuhao 觉得自己也没有完全想通。制造不对称性确实能显著提升 agent 在某一侧的能力，OSWorld 就是很好的例证，但这是不是长远来看最合适的路径，还有待观察。他认为 GUI 和 CLI 两条线完全可以并行发展，在数字世界里，两者的产物格式可以双向流动——agent 生成一半的，人类可以在 GUI 里继续编辑；人类做到一半的，agent 可以在 CLI 里接着完成。这种双向等价是 CLI anything 在设计上刻意追求的，它要求 agent 产出的格式和人类产出的格式在编译和结构上完全兼容。他还提到，社区里有人提出了一个有意思的需求：希望 agent 在用 CLI 操作的时候，能同时把渲染结果实时投射到显示器上供人监看——agent 不用 GUI 操作，但人可以盯着看。这种“内存级别的旁路渲染”让人类在不干扰 agent 工作流的情况下，随时了解当前状态，也是 GUI 理解能力在未来可以发挥作用的一个方向。八、自进化与 Eval 的难题Yuhao 提到他最近在和 Openclaw-RL 的作者交流，这个方向让他觉得很有潜力——在小规模上也能 work，梯度不会爆炸，确实能学到一些 style 上的改变。他的判断是：权重层面的进化是一定值得持续追求的，这可能占了模型成就的 60%~70%，剩下的才是 prompt 和 harness 的工程。但 self-evolution 最难的地方在于 Eval。如果沿用传统学术标准，就不可避免地需要引入人类评审或更强模型来做 judge，而自进化的核心就是要绕开这个。风格上的改变容易识别，行为模式和能力层面的深层改变很难量化。Yuhao提到 Lei Li 发表的观点——对 coding agent 最好的试金石不是 SWE-bench，而是自己放到 Claude Code 里用半小时一小时。他很同意这个说法，认为这本质上就是一种用户 AB test，只是工业界头部的团队才真正在做，学术界要怎么更方便地 access 到这种资源，是推动这个领域发展的一个重要问题。九、软件的未来CLI anything 接下来会怎么发展？Yuhao 的短期预期是：很多工作会像 coding 一样，在一两年内变成 vibe working——剪辑师、CAD 工程师、设计师会慢慢接过 CLI 产出的产物继续工作，随着模型更强、CLI anything 更完善，这条链条会越来越长，直到整个工作流都可以完全在 CLI 里完成。这和 Anthropic 最近发布的研究方向是一致的：AI 在很多行业的承载力其实已经很高，只是覆盖率还没跟上，而 CLI anything 是推动覆盖率提升的一个角色。对于软件服务商（“业主”）来说，CLI anything 提供的是一套相对温和的迁移路径——不需要大改现有系统，成本不高，却可以把自己的软件接进 agent 的工作流。Yuhao 认为这比 MCP 还要更朴素一点，门槛更低，因此对服务商来说有一定的吸引力。他也提到一个更激进的设想：如果把 CLI anything 和 AI OS 的概念结合起来，专门构建一个 agent-friendly 的平台，所有安装的软件都以 CLI 形式暴露接口，showcase 它能产出和人类等价的专业产物，然后让更多的生态和工作流迁移进来，影响会更大，风险也更大。AI 会吃掉软件吗？Yuhao 的答案是肯定的。他的出发点很直接：language 加多模态是人类最自然的交互介质，这件事本身就已经足够击穿很多人类最底层的需求。很多中老年用户宁愿和豆包对话，也不愿意用搜索引擎，这个趋势会延续。但同时强调，软件和服务本身不会消失。FFmpeg、游戏引擎、CAD 求解器这些底层工具依然会是不可缺少的基础设施，只是它们的入口会被 AI 统一替代。软件是越来越好的服务工具，而 AI 正在成为所有这些工具的统一入口。这个过程是一种熵减吗？AI 改造各行业，本质上是在把混乱的、碎片化的工作流和服务整合成更有序、更可操作的形式，CLI是可行且可靠的手段。我对这个未来非常乐观。
