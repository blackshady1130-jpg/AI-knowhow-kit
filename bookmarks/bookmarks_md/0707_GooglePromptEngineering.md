## 提示工程基础

### 什么是提示工程？

提示工程是设计高质量提示的过程，目的是引导大型语言模型（LLMs）生成准确的输出。这个过程涉及调整以找到最佳提示，优化提示长度，并评估提示的写作风格和结构与任务的关系。

记住 LLM 的工作原理：它是一个预测引擎。模型接收连续的文本作为输入，然后基于它所训练的数据预测下一个标记应该是什么。LLM 被操作化以重复这个过程，将先前预测的标记添加到连续文本的末尾，以预测下一个标记。

#### 提示工程流程

##### 设计提示

##### 调整参数

优化温度、Top-K、Top-P 等配置

##### 测试与迭代

分析输出并优化提示设计

##### 文档记录

### LLM 输出配置

#### 输出长度

控制响应中生成的标记数量。生成更多标记需要更多计算，可能导致更高的能源消耗、更慢的响应时间和更高的成本。

#### 温度 (Temperature)

控制标记选择中的随机性程度。较低的温度适用于期望更确定性响应的提示，较高的温度可能导致更多样化或意外的结果。

低温度 (0.1) 高温度 (1.0)

温度: 0.5

当前设置平衡了确定性和创造性

#### Top-K 和 Top-P

用于限制预测的下一个标记来自具有最高预测概率的标记。

##### Top-K 采样

选择模型预测分布中最可能的 K 个标记。较高的 Top-K 使模型输出更具创造性；较低的 Top-K 使输出更受限和事实性。

##### Top-P 采样 (核采样)

选择累积概率不超过某个值 (P) 的顶部标记。P 的值范围从 0（贪婪解码）到 1（LLM 词汇表中的所有标记）。

## 提示技术详解

🎯

### 零样本提示

Zero-shot Prompting

### 零样本提示

最简单的提示类型，只提供任务描述和一些文本让 LLM 开始。名称"零样本"代表"无示例"。

适用场景：简单明确的任务，如基本分类、总结或回答直接问题。

📋

### 单样本与少样本提示

One-shot & Few-shot

### 单样本与少样本提示

在提示中提供一个或多个示例，向模型展示要遵循的模式。这增加了模型遵循所需输出结构的可能性。

适用场景：需要特定格式输出的任务，或模型需要理解特定模式的情况。

⚙️

### 系统提示

System Prompting

### 系统提示

定义模型的整体背景和目的。设置"大局"，告诉模型应该做什么，例如翻译语言、分类评论等。

适用场景：需要模型执行特定任务或按特定方式格式化输出的情况。

🎭

### 角色提示

Role Prompting

### 角色提示

为语言模型分配特定的角色或身份。帮助模型生成与所分配角色及其相关知识和行为一致的响应。

适用场景：需要特定角色视角的响应，如旅游指南、教师或客户服务代表。

🧩

### 上下文提示

Contextual Prompting

### 上下文提示

提供与当前对话或任务相关的特定细节或背景信息。帮助模型理解所问内容的细微差别并相应地调整响应。

适用场景：需要针对特定情境或领域生成响应的情况。

🔗

### 思维链

Chain of Thought (CoT)

### 思维链

通过生成中间推理步骤来提高 LLMs 的推理能力，有助于 LLM 生成更准确的答案。

适用场景：需要逐步推理的复杂问题，如数学问题、逻辑谜题或多步骤任务。

### 高级提示技术比较

| 技术 | 主要特点 | 适用场景 | 优势 | 局限性 |
| --- | --- | --- | --- | --- |
| **后退提示**
(Step-back) | 先考虑与特定任务相关的一般问题，然后将答案输入到后续提示中 | 需要更广泛视角的复杂问题 | 激活相关背景知识，提供更全面的解决方案 | 需要额外的提示步骤，可能增加处理时间 |
| **思维链**
(Chain of Thought) | 生成中间推理步骤来提高推理能力 | 需要逐步推理的问题，如数学或逻辑 | 提高准确性，使推理过程可解释 | 生成更多标记，增加成本和时间 |
| **自一致性**
(Self-consistency) | 生成多样化的推理路径并选择最一致的答案 | 需要高准确度的复杂推理任务 | 提高响应的准确性和连贯性 | 成本高，需要多次生成 |
| **思维树**
(Tree of Thoughts) | 同时探索多条不同的推理路径 | 需要考虑多种可能性的复杂问题 | 适用于需要探索的任务，考虑多种方案 | 实现复杂，资源消耗大 |
| **ReAct**
(Reason & Act) | 结合自然语言推理和外部工具 | 需要与外部世界交互的复杂任务 | 能够检索信息并基于此采取行动 | 需要额外的API集成和工具访问 |

### 思维链 vs 思维树可视化比较

#### 思维链 (Chain of Thought)

思维链提示通过生成中间推理步骤来提高 LLMs 的推理能力。它遵循单一的推理线索，从初始问题到最终答案，像人类解决问题一样"思考"。这种方法对于数学问题、逻辑谜题或多步骤推理特别有效。

#### 思维树 (Tree of Thoughts)

思维树是思维链的泛化，允许 LLMs 同时探索多条不同的推理路径，而不仅仅是遵循单一的思维链。它维护一个思维树，每个思维代表一个连贯的语言序列，作为解决问题的中间步骤。这种方法特别适合需要探索的复杂任务。

### ReAct (Reason & Act) 工作流程

1

#### 推理 (Reason)

LLM 首先对问题进行推理并生成行动计划。它会分析问题，确定需要收集哪些信息，以及如何获取这些信息。

2

#### 行动 (Act)

LLM 执行计划中的行动，例如通过外部 API 或工具（如搜索引擎、代码解释器等）检索信息。

3

#### 观察 (Observe)

LLM 观察行动的结果，收集新获得的信息和数据，为下一步推理做准备。

4

#### 循环迭代

LLM 使用观察结果更新其推理并生成新的行动计划。这个过程持续进行，直到 LLM 达到问题的解决方案。

## 代码提示技术

### 代码提示概述

Gemini 主要关注基于文本的提示，这也包括编写返回代码的提示。作为开发人员，这可以帮助你加速编写代码的过程。

LLMs 可以在代码相关任务中提供巨大帮助，包括：

-   编写任意编程语言的代码
-   解释和分析现有代码
-   将代码从一种语言翻译成另一种语言
-   识别和修复代码中的错误
-   为代码生成文档

然而，请记住 LLMs 不能推理，它们重复训练数据，因此在使用生成的代码之前，必须仔细阅读和测试。

#### 代码提示类型

##### 编写代码的提示

生成新代码，从简单脚本到复杂功能

##### 解释代码的提示

分析和解释现有代码的功能和逻辑

##### 翻译代码的提示

将代码从一种编程语言转换为另一种

##### 调试和审查代码的提示

识别错误并提供修复方案

### 代码提示示例

#### 编写代码示例：Bash 文件重命名脚本

```
编写一个 Bash 代码片段，询问文件夹名称。
然后获取文件夹的内容并通过在文件名前加上 "draft" 来重命名里面的所有文件。
```

```
#!/bin/bash
# 询问文件夹名称
echo "输入文件夹名称："
read folder_name
# 检查文件夹是否存在
if [ ! -d "$folder_name" ]; then
    echo "文件夹不存在。"
    exit 1
fi
# 获取文件夹中的文件列表
files=( "$folder_name"/* )
# 通过在文件名前加上 "draft" 重命名每个文件
for file in "${files[@]}"; do
    new_file_name="draft_$(basename "$file")"
    mv "$file" "$folder_name/$new_file_name"
done
echo "文件重命名成功。"
```

```
解释以下 Bash 代码：
```

```
这个 Bash 脚本执行以下任务：

1. **用户输入**：
   - 它使用 echo 和 read 命令提示用户输入文件夹名称。输入的文件夹名称存储在 folder_name 变量中。

2. **文件夹存在检查**：
   - 它使用 [ ! -d "$folder_name" ] 条件检查指定名称的文件夹是否存在。如果文件夹不存在，它会显示错误消息（"文件夹不存在。"）并以退出代码 1 退出脚本。

3. **文件列表**：
   - 它使用 files=( "$folder_name"/* ) 命令列出指定文件夹中的所有文件。这会创建一个名为 files 的数组，其中包含文件夹中所有文件的路径。

4. **文件重命名**：
   - 它使用 for 循环遍历 files 数组中的每个文件。
   - 对于每个文件，它使用 new_file_name="draft_$(basename "$file")" 命令生成一个新的文件名。这个新文件名是通过在原始文件名前加上字符串 "draft_" 创建的。
   - 然后它使用 mv "$file" "$new_file_name" 命令重命名文件，将文件移动到新文件名。

5. **成功消息**：
   - 重命名所有文件后，它显示一条成功消息（"文件重命名成功。"）表明重命名操作成功。
```

## 最佳实践

找到合适的提示需要不断调整。Vertex AI 中的 Language Studio 是一个完美的地方，可以尝试各种提示，测试不同的模型。以下最佳实践将帮助您成为提示工程专家。

#### 提供示例

在提示中提供（单样本/少样本）示例是非常有效的，它充当了强大的教学工具。这些示例展示了期望的输出或类似的响应，允许模型从中学习并相应地调整自己的生成。

#### 简单设计

提示应该简洁、清晰，易于您和模型理解。作为经验法则，如果对您来说已经很混乱，那么对模型来说也可能很混乱。尽量不要使用复杂的语言，也不要提供不必要的信息。

#### 明确输出

具体说明所需的输出。简洁的指令可能不足以引导 LLM 或太过笼统。在提示中提供特定细节（通过系统或上下文提示）可以帮助模型集中注意力于相关内容，提高整体准确性。

#### 使用指令而非约束

研究表明，在提示中专注于积极指令比严重依赖约束更有效。指令直接传达期望的结果，而约束可能会让模型猜测什么是允许的。尽可能使用积极指令：告诉模型应该做什么，而不是不应该做什么。

#### 控制最大标记长度

通过在配置中设置最大标记限制或在提示中明确请求特定长度来控制生成的 LLM 响应的长度。例如："用一条推文长度的消息解释量子物理学。"

#### 在提示中使用变量

在提示中使用变量，可以为不同的输入而更改，使提示更加动态和可重用。变量可以节省您的时间和精力，避免重复自己。这在将提示集成到自己的应用程序中时特别有意义。

#### 实验不同的输入格式和写作风格

不同的模型、模型配置、提示格式、词语选择和提交可能产生不同的结果。因此，重要的是要尝试提示属性，如风格、词语选择和提示类型（零样本、少样本、系统提示）。

#### 适应模型更新

了解模型架构变化、添加的数据和功能。尝试更新的模型版本并调整提示以更好地利用新模型功能。像 Vertex AI Studio 这样的工具非常适合存储、测试和记录提示的各种版本。

#### 实验不同的输出格式

除了提示输入格式外，考虑尝试输出格式。对于非创意任务（如提取、选择、解析、排序、排名或分类数据），尝试将输出返回为结构化格式，如 JSON 或 XML。这有助于限制幻觉并提供更一致的结果。

### 记录各种提示尝试

提示输出可能在不同模型、不同采样设置甚至同一模型的不同版本之间有所不同。此外，即使对同一模型的相同提示，输出句子格式和词语选择的微小差异也可能发生。

我们建议创建一个详细记录提示尝试的表格，这样您可以随着时间的推移了解哪些方法有效，哪些无效。这种方法的优点是，当您不可避免地不得不重新访问您的提示工作时，您有一个完整的记录。

| 名称 | 目标 | 模型 | 温度 | Top-K | Top-P |
| --- | --- | --- | --- | --- | --- |
| movie\_classification\_v1 | 对电影评论分类为积极、中立或消极 | gemini-pro | 0.1 | N/A | 1 |
| movie\_classification\_v2 | 对电影评论分类并返回 JSON 格式 | gemini-pro | 0.1 | N/A | 1 |
| pizza\_order\_parser\_v1 | 将披萨订单解析为 JSON | gemini-pro | 0.1 | N/A | 1 |

提示工程是一个迭代过程。设计和测试不同的提示，分析和记录结果。根据模型的表现完善您的提示。继续实验，直到您获得期望的输出。当您更改模型或模型配置时，请返回并继续尝试之前使用的提示。

### 提示开发过程

1

#### 明确目标

明确定义您想要 LLM 执行的任务。确定任务类型（如分类、生成、提取等）和您期望的输出格式和质量。

2

#### 设计初始提示

基于最佳实践创建初始提示。根据任务的复杂性，选择合适的提示技术（零样本、少样本、系统提示等）。

3

#### 测试与调整

使用各种输入测试提示。观察输出，分析模型的表现和局限性。调整提示语言、结构和示例，并微调模型参数。

4

#### 记录与评估

详细记录每个提示版本及其结果。评估提示在各种情况下的性能，包括边缘情况和异常输入。

5

#### 优化与部署

基于评估结果优化提示。准备将最终提示集成到应用程序中。建立监控系统，跟踪生产环境中的表现。

## 总结与延伸

《Prompt Engineering》白皮书全面讨论了提示工程的各个方面，从基础技术到高级策略。它强调了提示工程是一个迭代过程，需要不断实验和改进。

提示工程不仅仅是一种技术，更是一种艺术，需要创造力、分析思维和对细节的关注。通过掌握本指南中介绍的各种技术和最佳实践，您将能够更有效地利用大型语言模型的潜力。

#### 实践建议

-   建立提示库：创建并维护一个提示库，包含不同任务和场景的有效提示
-   定期测试：随着模型的更新，定期测试您的提示效果
-   社区参与：参与提示工程社区，分享经验和学习新技术
-   跨模型测试：在不同的 LLM 上测试您的提示，了解它们的差异和共性

记住，最佳提示是通过实践和反复试验发现的。随着您经验的积累，您将开发出越来越有效的提示策略，能够充分发挥 LLM 的潜力。
