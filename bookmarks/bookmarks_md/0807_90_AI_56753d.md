# 当效果调优的90%都可以由AI完成

本文呼应我去年写的99.9%的代码AI生成率那篇文章。我最近的AI Coding实践案例，谈当下的99.9% AI生成率如果说那篇文章标志着前端后端开发的工作开始被实质性变更，那么本文则标志着算法类岗位的工作也开始被实质性变更。当然本文说的算法岗并非仅限互联网大厂中调模型的稀少岗位，而是指所有需要调优效果的岗位，事先并不知道要做到什么程度才能达到什么样的效果。其实我本想等到本文的产出发布之后再写这篇文章的。但无奈我收到了Claude额度的强制CD，所以只能考虑先利用这段空闲来写此文。我已经买了三个Claude Max x20账号，正在弄第四个账号，但被苹果风控导致第四个号的到位又被强制CD 48h。1、不可量化任务的AutoResearchAutoResearch这个想法虽然是很多人都独立发现，但最终流行的还是Andrej Karpathy的实践。 Andrej Karpathy 的 AutoResearch 尝试 他的实践是用来优化LLM模型结构，有着很明确的优化目标。不过当你细看他的方式时会发现他仍然加了不少不太能量化的目标，例如实现复杂度等。但实际上AutoResearch的方案并不需要有一个量化的目标，只需要Agent能够根据目标比较两个版本方案的结果的好坏，并能思考后续改进方向即可。我把这称之为“语义质量比较函数”和“语义梯度”。当然随着模型能力的不断提升，更精确描述模型的能力的表述也变得越来越抽象，已经让不少人很难直接理解了。证明一件事存在的最好方式就是做一个实例，虽然仍然很难传达这个抽象的视角，但至少能够让信息接收方认知到确实存在一些不符合他原本想法的案例，不会早早放弃思考。所以我做了一个解读新知类书籍的方案，并使用AI进行自己优化。从结果来看，目前实现的效果已经超过了我原本的预期，虽然对我来说谈不上很意外的惊喜，但确实验证了可以做，而且可以做得不错。整个流程中所有需要效果调优的环节都是靠AI完成的，我和一些人对生成的结果进行了一些人工评价，但整体的效果调优工作基本都是AI完成的，分配一下credit的话，AI至少承担了90%的工作量和价值产出。整个工作包括书的内容拆解和解读撰写、图书选择、专栏头像、专栏开篇词、TTS音色调教、高质量TTS方案、播客前后音乐、封面图、视频静态配图等，都是使用该方式完成。当然后续很多环节并没有投入太多的时间预算，但确实不是之前靠人去调的模式了。当然说再多都没啥用，最终应该看实际成品效果。我尽量在本周先把这个专栏的第一期做出来供读者观察其质量。当然这个内容生产的结果就是一个有价值的事情，希望它也能得到足够的关注。在此之前就不剧透太多了。2、管中窥豹本文并不会太详细的介绍整个过程，但会少量地讲一些不那么常见的信息。整个过程是采用更加Agentic的架构来实现的，具体来说是我在 Claude Code -p 成为智能函数 中描述的这种方式作为原子能力来进行整个流程的构建。由于Claude 1M Context window做这个任务仍然捉襟见肘，所以我在Claude -p之上还是构建了一个更固化的workflow来实现最核心流程，而其中的节点是claude agent。1M Context不够的原因是本项目的调教目标是高质量。但这也导致这个方案调优成本高得吓人。即使是比较勤奋的人，能用满一个Claude Max x20账号的额度就已经算不轻松了。但当采用我这样的方式来做方案调优时，由于单个任务的巨大token消耗，Claude Max订阅的额度就在被疯狂消耗。从第一个账号到开第二个，再到第三个、第四个，中间的过程在缩短。从事后估计来看，我大概在半周的时间用用完了前两个Max x20订阅的周额度，这导致我不得不考虑开第三个和第四个账号。可以说整个过程的瓶颈都是在token补充方面。其中音频方面的迭代还是需要人比较多，特别是基准音色的选择和比较上，我还是投入了不少时间。目前的模型对于音频内容的输入能力仍然不普及，能力也算不上很优秀。另外，本文与Harness无关，虽然可能有人会觉得有关，但对我来说相关性太低，近似等同于正交。最后放一个过程中效果供参考，不代表最终发布质量。交流与合作我目前正在看机会，详情请见 年底的逆向招聘尝试 2025.11如果希望和我交流讨论，或参与相关的讨论群，或者建立合作，请加微信，联系方式请点击 -> 专栏简介 及 联系方式 2024。本文于2026.3.31 首发于微信公众号。
