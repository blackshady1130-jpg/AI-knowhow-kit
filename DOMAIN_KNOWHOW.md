## 2) 核心判断范式（提炼）

### A. 问题定义优先

- 先定义业务目标与边界，再讨论模型能力与方案。
- 反对“技术先行、场景后补”的倒置流程。

### B. 闭环优先于单点优化

- 偏好完整链路：`数据 -> 训练 -> 评测 -> 应用 -> 反馈数据`。
- 强调机制持续性，不把一次指标抬升当作长期能力提升。

### C. 评测是治理工具，不是装饰分数

- 关注评测设计质量（任务定义、样本、口径、对照、外推边界）。
- 重视 in-domain 与 out-of-domain 的拆分，避免误判泛化能力。
- 强调 badcase 积累和迭代验证。

### D. Agent 是系统工程

- 关注 workflow、tooling、runtime、context/memory，而非只看模型分数。
- 对“结构化编排”有长期兴趣：skills、MCP、上下文工程、执行稳定性。

### E. 商业价值必须可量化

- 偏好“效率价值/ROI/可复用资产”叙事，而非纯概念共识。
- 对 SaaS、AI 产品增长、组织能力与交付链路有持续观察。

## 3) 场景化方法论（可直接复用）

### 模型合作六步法

1. 定义问题：任务目标、边界、风险阈值。  
2. 定义评测：指标口径、基线、样本结构。  
3. 定义数据：来源、质量标准、标注协议。  
4. 定义训练：对应 pretrain/SFT/RLHF 诉求。  
5. 定义上线：ToB/ToC 场景与交付形态。  
6. 定义迭代：badcase 回流、复评节奏、验收机制。  

### 报告/数据阅读四步法（来自博客沉淀）

1. 看利益结构：谁出资、谁受益。  
2. 看方法支撑：样本、变量、推导链。  
3. 看口径与偏差：是否“绕弯论证”或选择性呈现。  
4. 看适用边界：哪些结论可迁移，哪些不可外推。  

## 4) 稳定风险观

- 对“迎合式回复/谄媚”高度敏感。
- 对“高分低可用”高度敏感。
- 对“概念堆叠但无验证机制”高度敏感。

## 5) 给 Agent 的判断句式

- “先明确任务口径，再讨论模型优劣。”
- “如果只有 in-domain 提升，暂不判定为泛化跃迁。”
- “没有评测闭环的优化，默认视为一次性改进。”
- “先给最小可落地方案，再给扩展路线图。”

## 6) 行业扫描样本边界与主题簇（v0.3）

### 样本边界

- 当前行业扫描样本属于“高信噪观察集”，规模约 180+ 条，覆盖论文、观点文章、行业动态三类来源。
- 时间窗口以近一年为主，适合做近期趋势与策略判断，不适合外推长期周期结论。
- 内容来源中中英文并存，但中文二手解读占比不低，对原始数据口径需二次核验。

### 主题簇（用于快速归类）

1. 公司与竞争格局（头部厂商产品与组织动作）  
2. 模型能力与训练范式（预训练/SFT/RL/mid-training）  
3. Agent 与上下文工程（workflow、tool use、memory、runtime）  
4. 评测与 benchmark（任务定义、样本构成、口径和可解释性）  
5. 商业化与 SaaS（ROI、留存、交付链路、组织效率）  
6. 安全与治理（风险使用、政策约束、合规挑战）  

### 判断边界

- 行业扫描用于“形成可验证假设”，不是统计全景，不代表市场全样本分布。
- 观点类材料可用于方向判断，但不能替代可复现实验或经营数据。
- 任何强结论都应补充适用前提、失效条件和后续验证动作。
