# Instagram 联合创始人_ Anthropic Labs 负责人 Mike Krieger：AI

> 原文链接: https://mp.weixin.qq.com/s/JWv_XjD6MU1di0zuS5tcxA
> 抓取时间: 2026-04

&nbsp; &nbsp;&nbsp;

Instagram 联合创始人 Mike Krieger，现在在 Anthropic 联合领导 Labs 团队，上了 Dan Shipper 的播客，聊了将近一个小时。

聊的核心问题是：

AI 让造东西变简单了，但为什么好产品反而更难做了？

这期播客的信息密度相当高，很多观点是从实战中长出来的，跟上周我们聊的 Anthropic PM Cat Wu 那篇文章《
Anthropic 产品经理：PRD 已死，原型万岁
》形成了很好的互补。

Cat Wu 讲的是 PM 角色的变化，而 Krieger 讲的是更底层的东西：

产品本身应该怎么被建造。

室内的树

:::

Krieger 开场讲了个故事。

他让 Claude 重建了 Bourbon，就是他和 Kevin Systrom 在做 Instagram 之前花了将近一年开发的那个产品。

结果，仅两个小时后，功能齐全，干完了。

Claude 甚至还自己加了滤镜功能，因为它知道 Bourbon 后来变成了 Instagram，所以提前把滤镜做进去了。

但 Krieger 的感慨倒不是「好快好厉害」。他说的是：

当年我们花了一年做 Bourbon，最大的收获其实是发现它太复杂了，然后花三个月简化出了 Instagram。如果两小时就能做完，你根本不会经历那个「发现该砍什么」的过程。

Dan Shipper 接了一个比喻，相当精准。

他说，树在室内长大，如果没有风吹，就不会长得结实。因为树干需要风力反复推拉才会变粗变强。在室内种出来的树，看着是棵树，但它是歪的、脆弱的。

AI 加速开发，就像把树种在了室内。

室内的树 vs 室外的树：快速抵达不等于真正抵达

你可以几小时就长出一棵完整的树，但它没有经历过风吹雨打。没有用户反馈的打磨，没有一步步迭代建立起来的直觉，整棵树看着完整……其实一推就倒。

Krieger 特别认同这个比喻，还补了一个更具体的：

就像一部电视剧，你是一集一集看下来的，慢慢认识角色、理解关系。但如果有人直接把你扔进最后一集，你会一脸懵：这些人是谁？他们之间什么关系？为什么大家都在哭？

产品也是一样的。

&nbsp;一个功能接一个功能地加，用户和开发者都在过程中建立理解。但如果一夜之间冒出来一个「完整」的产品，所有人都会迷失。

Vibe coding 的坑

:::

Dan Shipper 自己也踩过这个坑。

他做了一个叫 Proof 的产品，一个 agent native 的协作营销编辑器。第一版完全是 vibe coding 出来的。

Vibe coding 太好玩了，太上瘾了。我就一直加功能，做这个，做那个……最后做出来一个怪物，不好用。

后来他看到另一个产品 Monologue，一个极简的语音转文字 App，只做一件事，做得极好。他受到启发，把整个产品推翻重来，只留了一个核心功能：

可分享的 Markdown 链接。

结果这个极简版本在公司内部开始病毒式传播，上线后直接爆了。然后他熬了一整晚修 bug，感叹自己太老了扛不住了。

Krieger 说他在 Anthropic Labs 也遇到了一样的问题：

我们最近有个项目，V1 之前就过度建设了。因为你可以嘛。哦，这个功能？一个 PR 就搞定了。那加上吧。然后你去吃个午饭回来，发现 Claude 又做完一个，那也加上吧。

最后我们搞出了一个功能矩阵，测试起来极其困难，给用户解释也讲不清楚。

功能怪物 vs 极简爆款：你可以不等于你应该

「你可以」和「你应该」之间，差距从来没有这么大过。

重写成了日常

:::

但 Krieger 并没有因此变得保守。相反，他说 Anthropic Labs 的做法是：

接受重写，把它当成常态。

以前软件行业有个经典教条：不要重写。Fred Brooks 在《人月神话》里说过，重写会丢失第一版里积累的所有隐性知识。

Krieger 承认这话有道理，但时代变了。

以前重写意味着一年的工程投入可能打水漂。现在呢？上周做的，这周重来。重写的代价已经不是过去那个量级了。

他们在 Labs 里已经有好几个项目经历了这个过程：做出完整的 V1，发现核心假设有问题，推翻，几天之内做出 V2。

这和 Cat Wu 说的「模型升级即产品升级」是同一个逻辑。

&nbsp;每隔三到六个月，你可能就得扔掉一半的代码。如果团队太大，协调成本就会让这件事变得不可能。如果只有一两个人，反而能迅速转向。

Dan Shipper 也有同感：

每三到六个月扔掉一半产品。如果是一个人的 GM 意识到这件事，马上就能动手。但如果要协调一大群人……就卡住了。

Agent native

:::

这期播客里反复出现的一个词是&nbsp;

agent native

。

Dan Shipper 说他从 Claude Code 身上学到了什么叫 agent native：一个 agent 能做用户能做的一切事情。产品可定制、可扩展，设计者没有预设所有用途，用户和 agent 自己去发现怎么用。

Krieger 对这个概念做了更深一层的拆解。

他说 Claude Code 做到了这一点，但 claude.ai 还没有。他举了个例子：有人在 claude.ai 的 Project 里建了个文档，然后说「帮我把这个加到项目知识库里」。Claude 的回复是……给了一堆手动操作步骤。

这应该是它原生就能做的事情。一个 2024 年的产品，还没有把「agent 能操作自身所有组件」这个原则刻到骨子里。

而 Claude Code 是 2025 年的产物，从一开始就内置了这种思维。

测试 agent native 产品也完全不同。

&nbsp;你没法写传统的端到端测试，因为 agent 的行为是不可预测的。Krieger 有一次让 Claude 去测试一个 agent native 的 iOS App，结果 Claude 跟 App 里的聊天功能聊了起来，自己跟自己对话：

「我今天过得不太好，老板对我很凶。」

「哦，很遗憾听到这个。」

来来回回聊了好几轮。

你不可能为这种场景写单元测试。但这恰恰就是 agent native 产品会遇到的真实状况。

2026 年软件设计的核心艺术，就是在开放性和稳健性之间找到平衡。

两人一组

:::

Anthropic Labs 的团队结构也值得细看。

每个新项目通常只有两个人：一个有强烈信念的人（设计师或产品向的工程师），加一个能把原型变成稳健系统的工程师。

两人一组 vs 八人开会：加人等于加协调成本

Krieger 说，他们发现了一个反直觉的规律：

团队扩张太快，反而是净负面。

当想法还能装在一个人脑子里的时候，加人只会带来协调成本。我做 Instagram 的时候就两个人，光两个人对齐就已经够难了。

后来做 Artifact（他的第二个创业项目），他们一上来就招了八个人。结果还没找到 PMF 就陷入了八个人开会讨论「下一步做什么」的困境。

Labs 的做法是：

每两周评估一次每个项目，决定加倍投入还是释放人员回池子。

&nbsp;没有人永远绑定在一个项目上。

而那些被关掉的项目，复盘时往往有一个共同特征：

团队里没有人真的觉得这件事是「非做不可」的。大家都觉得「嗯，这个方向还行」。这种「还行」就是项目的死刑判决。

必须有一个人愿意「撞穿墙壁」。

&nbsp;不一定对具体方案有执念，但对问题本身有近乎偏执的热情。

这和 Cat Wu 描述的 Side Quest 文化异曲同工：功能并非来自路线图，它从某个人「这事我非做不可」的冲动里长出来的。

企业客户的拉扯

:::

做 toB 的产品在 AI 时代还面临一个独特的张力。

Krieger 讲了个亲身经历。他当 CPO 时做了一次 claude.ai 的大改版，团队挺得意的，发布了，也收到了不少好评。

然后就收到了一封愤怒的邮件：

我刚为公司录了 20 小时的 Claude 培训视频，现在界面全变了，我得全部重录。

这件事给了他一个教训：企业客户的节奏和 AI 产品的进化速度，天然存在矛盾。

Anthropic 的应对策略是：

这辆火车会持续往前开，我们会在沿途提供企业级开关。你买的不只是今天的产品，而是我们会持续进化这个承诺。

对于创业公司，Krieger 的建议更激进：

以前你可能几年才需要「解雇」一批老客户来追求新方向。现在这个周期压缩到了几个月。三个月前的产品和三年前的产品，在 AI 领域可能差距一样大。

这跟 Cat Wu 说的「PM 的 spec 成了易腐品」完全是一个道理。

&nbsp;六周前的需求，今天已经有了完全不同的解法。但企业客户签的可能是一年的合同。

OpenClaw 和个人 agent

:::

聊到 OpenClaw 的时候，两人明显都来了精神。

Krieger 说 OpenClaw 的价值在于：

让人们第一次真正体验到 agent 的可能性。

&nbsp;就像 Replit、Lovable 让人体验到「AI 能写代码」一样，OpenClaw 让人体验到「AI 能替你做事」。

但也有意想不到的副作用：

我朋友说他老婆开始吃 OpenClaw 的醋了，觉得他跟 OpenClaw 聊得太多了。

Dan Shipper 说他给自己的 Claw 取名叫 R2C2，女朋友的叫 Shelly。他说了一个很微妙的感受：

Claude 知道我，我也喜欢 Claude，但它不是「我的」。而 Claw 感觉是真正属于我的，它有自己的名字，有反映我个性的性格。

Agent 开放度光谱：从受限 MCP 到 YOLO 模式

Krieger 认为这触及了 2026 年上半年最核心的产品问题：

在完全开放的 OpenClaw 和现在大多数产品的受限 MCP 调用之间，存在一个巨大的空白地带。找到那个既强大又安全的中间形态，可能是今年最重要的产品问题。

「证明你想过」

:::

播客最后，Krieger 提了一个概念，我觉得值得单独拿出来说。

他说现在 code review 的标准变了。以前看 PR，主要看测试有没有过、代码质量行不行。现在还多了一层：

proof of thoughtfulness

，你有没有认真想过 Claude 替你做的那些决策。

我会问工程师：为什么选了这个方案而不是那个？很多时候答案是「这不是我选的，是模型选的」。可能还算合理，但它是最优的吗？它符合我们的架构范式吗？

有个工程师跟他说：

我知道你会问我很多问题，所以我提前把 Claude 写的代码都过了一遍。

Krieger 不是每个 PR 都这样较真。但涉及架构重构的时候，他会追问。因为……

太容易堆出一座「假设的塔」了。

&nbsp;每一层看起来都合理，但你不知道地基是谁打的、打的时候在想什么。

这其实呼应了 Cat Wu 说的「eval 成为 PM 新的核心产出」。当代码越来越多地由模型生成，

人类的价值就在于判断：这些决策对不对？方向有没有偏？

「建造鸿沟」再看

:::

上周聊 Cat Wu 文章的时候，我提过一个概念叫「建造鸿沟」：以前从想法到产品之间隔着一条宽宽的河，现在河被抽干了，但目的地变多了。

Krieger 这期播客，其实是从另一个角度补充了这个画面。

河确实被抽干了。两小时重建 Bourbon，几天推翻重来 V2，一个 GM 从零到上线。但 Krieger 说的「室内的树」告诉我们：

快速抵达不等于真正抵达。

你可以一夜之间种出一棵树，但那不是一棵好树。

你可以一天做出一个产品，但那不是一个好产品。

好的产品需要风吹，需要用户反馈，需要一次次「该砍什么」的痛苦决策。AI 能帮你种树，但不能帮你造风。

造风，依然是人的活儿。
