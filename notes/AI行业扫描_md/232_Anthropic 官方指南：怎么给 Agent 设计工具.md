*Learn how the Claude Code team designs, tests, and evolves tools by thinking from the model's point of view.*

---

构建 agent harness 最难的部分之一，就是设计它的工具（tools）。

Claude 完全通过工具调用（tool calling）来行动，但在 Claude API 中，工具可以通过 `bash`、`skills`、`code execution` 等 primitives 以多种方式构建。

那么，你该如何设计 agent 的工具？是给它一个通用工具（比如 bash 或 code execution），还是五十个专用工具，每个用例一个？

要站在模型的角度思考，想象一下你被给了一道难数学题。你想要什么工具来解决它？这取决于你自己的技能水平！

纸和笔是最低配置，但你会受到手动计算的限制。计算器会更好，但你需要知道如何操作更高级的选项。最快最强大的选项是电脑，但你必须知道如何用它来编写和执行代码。

这是设计 agent 的一个有用框架：**你想给它与自身能力相匹配的工具。** 但你怎么知道它的能力是什么？你要仔细观察，阅读它的输出，进行实验。你要学会像 agent 一样思考。

如果你在构建 agent，你会面临和我们一样的问题：何时添加工具，何时删除工具，以及如何区分这两种情况。以下是我们在构建 Claude Code 时如何回答这些问题的，包括我们最初犯错的地方。

---

## 用 AskUserQuestion 工具改善信息获取

在构建 AskUserQuestion 工具时，我们的目标是提升 Claude 提问的能力（通常称为 elicitation）。

虽然 Claude 可以直接用纯文本提问，但我们发现回答这些问题感觉花了不必要的时间。如何降低这种摩擦，增加用户和 Claude 之间的沟通带宽？

### 尝试一：修改 ExitPlanTool

我们尝试的第一种方法是在 ExitPlanTool 中添加一个参数，在计划旁边附带一组问题。这是最容易实现的修复，但它让 Claude 感到困惑，因为我们同时要求它提供计划和关于计划的问题。如果用户的回答与计划内容相冲突怎么办？Claude 需要调用两次 ExitPlanTool 吗？我们知道这个策略行不通，所以回到了起点。

### 尝试二：更改输出格式

接下来，我们尝试更新 Claude 的输出指令，让它使用一种稍微修改过的 markdown 格式来提问。例如，我们可以要求它输出一个带有括号中备选项的项目符号问题列表，然后我们可以解析并将该问题格式化为用户界面。

Claude 通常可以产生这种格式，但不够可靠。它会附加额外的句子，丢弃选项，或者完全放弃这种结构。继续尝试下一种方法。

### 尝试三：AskUserQuestion 工具

最终，我们决定创建一个 Claude 可以在任何时候调用的工具，但特别提示它在计划模式下这样做。当工具触发时，我们会显示一个模态框来展示问题，并阻塞 agent 的循环直到用户回答。

这个工具允许我们提示 Claude 产生结构化输出，并帮助我们确保 Claude 给用户提供多个选项。它还给用户提供了组合这个功能的方式，例如在 Agent SDK 中调用它或在 skills 中引用它。

最重要的是，Claude 似乎喜欢调用这个工具，我们发现它的输出效果很好。毕竟，即使是设计最好的工具，如果 Claude 不理解如何调用它，也是无用的。

这是 Claude Code 中 elicitation 的最终形态吗？我们怀疑不是。随着 Claude 变得更有能力，服务于它的工具也必须进化。下一节展示了一个曾经有帮助的工具开始妨碍工作的案例。

---

## 随能力更新：任务与待办事项

当我们最初发布 Claude Code 时，我们意识到模型需要一个**待办事项列表**来保持正轨。待办事项可以在开始时写下，随着模型完成工作而被勾选。为此，我们给了 Claude `TodoWrite` 工具，它可以写入或更新待办事项并将其显示给用户。

但即便如此，我们经常看到 Claude 忘记它必须做什么。为了适应，我们每 5 轮插入系统提醒，提醒 Claude 它的目标。

随着模型改进，它们发现待办事项列表有限制性。被发送待办事项列表的提醒让 Claude 认为它必须坚持列表，而不是在意识到需要改变方向时修改它。我们还看到 Opus 4.5 在使用子 agent 方面变得更好，但子 agent 如何在共享的待办事项列表上协调？

看到这一点，我们用 **Task 工具**替换了 TodoWrite 功能。待办事项专注于让模型保持正轨，而任务帮助 agent 相互沟通。任务可以包含依赖关系，在子 agent 之间共享更新，模型可以修改和删除它们。

**随着模型能力的提升，你的模型曾经需要的工具现在可能正在限制它们。** 不断重新审视关于需要哪些工具的先前假设非常重要。这也是为什么坚持支持一小组具有相当相似能力特征的模型是有用的。

## 设计搜索接口

我们构建的最重要的工具是那些让 Claude 能够自己找到上下文的工具。

当 Claude Code 最初在内部发布时，我们使用了 RAG：一个向量数据库会预先索引代码库，harness 会在每次响应之前检索相关片段并将其交给 Claude。虽然 RAG 强大且快速，但它需要索引和设置，并且在各种不同环境中可能很脆弱。最重要的是，Claude 是被**给予**这个上下文，而不是自己找到上下文。

但如果 Claude 可以在网上搜索，为什么它不能搜索你的代码库？通过给 Claude 一个 Grep 工具，我们可以让它搜索文件并自己构建上下文。

**随着 Claude 变得更聪明，在给予正确工具的情况下，它在构建自己的上下文方面变得越来越好。**

当我们引入 Agent Skills 时，我们正式确立了**渐进式披露（Progressive Disclosure）**的理念，允许 agent 通过探索逐步发现相关上下文。

Claude 现在可以读取 skill 文件，这些文件可以引用模型可以递归读取的其他文件。事实上，skills 的一个常见用途是为 Claude 添加更多搜索能力，比如给它关于如何使用 API 或查询数据库的指令。

在一年多的时间里，Claude 从几乎无法构建自己的上下文，发展到能够在多层文件中进行嵌套搜索，找到它所需的确切上下文。

渐进式披露现在是我们用来在不添加工具的情况下添加新功能的常用技术。

---

## 渐进式披露：Claude Code Guide Agent

Claude Code 目前有约 20 个工具，我们的团队经常重新审视是否需要所有这些工具才能让 Claude 最有效。添加新工具的门槛很高，因为这给模型增加了一个需要考虑的选项。

例如，我们注意到 Claude 对如何使用 Claude Code 了解不够。如果你问它如何添加 MCP 或斜杠命令是做什么的，它无法回答。

我们可以把所有这些信息放在系统提示中，但考虑到用户很少问这些问题，这会增加上下文污染并干扰 Claude Code 的主要工作：编写代码。

相反，我们尝试了渐进式披露：我们给了 Claude 一个指向其文档的链接，它可以在需要时加载和搜索。这有效，但 Claude 会将大量文档拉入上下文来找到用户本可以用一句话得到的答案。

所以我们构建了 **Claude Code Guide**——一个 Claude 在用户询问 Claude Code 本身时调用的子 agent。子 agent 在自己的上下文中进行文档搜索，遵循关于如何搜索和提取什么的详细指令，并只返回答案。主 agent 的上下文保持干净。

虽然这不是完美的解决方案（当你问它如何设置自己时，Claude 仍然可能感到困惑），但我们能够在不添加新工具的情况下扩展 Claude 的行动空间。

## 像 Agent 一样思考是一门艺术，而非科学

为你的模型设计工具既是一门艺术，也是一门科学。它在很大程度上取决于你使用的模型、agent 的目标以及它运行的环境。

**我们最好的建议？经常实验，阅读你的输出，尝试新事物。最重要的是，尝试像 agent 一样思考。**
