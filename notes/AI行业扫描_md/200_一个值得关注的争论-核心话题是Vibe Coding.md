# 一个值得关注的争论（核心话题：Vibe Coding）

> 原文链接: https://x.com/i/grok/share/c152d0cd13b94e3c904142a8786a6fc6
> 抓取时间: 2026-04

他们正在激烈争论一个核心话题：

在「Vibe Coding」（AI驱动的"氛围式/代理式编码"时代），代码到底还需不需要按传统软件工程原则拆分成多个小文件、控制行数、强调"人类可维护性"？还是应该大胆把逻辑全塞到一个超级大文件里，让AI代理更高效地迭代？

## 上文系列讨论的脉络（从根源开始）

**起源：**

@yetone 先发了一张13000行大文件的截图（挂人），然后直接发了这条引发一切的推文：

"Vibe Coding 时代，代码行数已经是不需要在意的事情了，放在一个文件中反而更好，减少 turns"

**他之前的相关观点（同一波讨论链）：**

- 早就去掉了所有 linter/formatter（因为只增加 token 消耗，减慢迭代）。
- 认为程序员在 Vibe Coding 里的核心价值是"steerability-skill"（用好奇心引导AI问根因、边缘case等）。
- 今天最新补充：不是完全反对拆分，但只在有复用逻辑或有状态机/状态流转时才需要拆，否则"拆了反而浪费LLM心智去维护架构"。

整个上文就是在反复论证：传统"人类最佳实践"（模块化、拆文件、保持可读）在AI代理时代反而成了拖累。

## 这个具体推文的位置和触发

有人回复他："面试这么说直接完蛋，请问怎么维护，他们不听的。"

@yetone 直接怼回去（就是你链接的这条）：

"所以不要面试，而且这种东西为什么需要人来维护呢？提出怎么维护的人，基本上这个公司就完了"

**一句话总结他的立场：**

如果一个公司还在纠结"人怎么维护AI生成的代码"，说明它还没进入Vibe Coding时代，基本已经完了。代码就是给AI看的，人类别瞎操心。

## 这个推文下面的回复讨论（精华总结）

回复分成两大阵营，吵得很真实：

### 支持/附和 yetone 的（偏激进派）：

- "我也偷偷写到一个文件去😌"
- "昨天按人类最佳实践拆文件后，AI一直在读啊读浪费token，以后不干了"
- 认可他发的详细对比：拆文件 = LLM 要多做规划 + 多轮 Grep+Read+MultiEdit（心智负担爆炸）；单文件 = 一次 Grep + 一次 Read + 一次 Edit（极简）。
- 最新补充：Reviewer Agent 看不下去再让它拆就行了，人别管。

### 反对/质疑的（传统派/务实派）：

- "系统报警时还是得找 code owner，人不能真等AI来搞黑盒吧"
- "AI是持续熵增，不考虑维护迟早药丸；而且要担责，谁来背锅？"
- "大文件 tracing 效率低，LLM 也容易在超长文件里迷路"
- "完全不考虑可维护性了？大型项目后期人介入会很难受，只能一条路走到黑"
- 有人调侃总结："所有代码放一个文件，不然公司就完蛋了😂"

### yetone 的反击风格：

每次都甩出那条经典长回复（你回复里也反复出现），核心三点：

1. 拆文件本身就需要LLM先花心智规划 + 以后每次迭代都要防崩。
2. 多文件修改 = 多轮工具调用（每个调用都是LLM巨大心智消耗）。
3. 没复用的话，拆出去干嘛？

他还强调：Read tool 有窗口限制，Grep 一次就能定位，不会真的把整个大文件全吃进上下文。

## 整体结论

这不是简单的"文件大小"之争，而是编程范式彻底转向的意识形态冲突：

- **旧世界**：代码写给人看 → 模块化、可读性、面试问维护、人类担责。
- **新世界（yetone 推的）**：代码主要写给AI代理看 → 单文件极致上下文、极简工具调用、人类只负责"vibe 和 steer"、维护交给下一代更强模型。

目前讨论还没结束，但已经从一个具体大文件例子，延伸到"公司是否该面试传统维护题""AI时代谁来担责""LLM工具调用到底多大成本"这些根本问题。双方都拿出真实使用经验在对线，挺有营养的辩论。
